# 기여 가이드

## 이슈 작성

1. 기존 이슈를 검색해 중복 여부를 확인합니다.
2. 목적에 맞는 이슈 템플릿을 사용합니다.
3. 여러 저장소에 영향을 주면 관련 저장소, 이슈, PR과 예상 병합 순서를 함께 적습니다.
4. 로그와 화면 캡처에서 토큰, 개인정보, 내부 URL 등 민감 정보를 제거합니다.

보안 취약점은 공개 이슈로 등록하지 마세요. 저장소에서 안내하는 비공개 보안 제보 채널을 사용합니다.

## Linear 이슈와 브랜치

1. 작업을 시작하기 전에 Linear 이슈가 있는지 확인합니다.
2. Linear 이슈에서 **Copy git branch name**을 사용합니다. 단축키는 `Cmd/Ctrl` + `Shift` + `.`입니다.
3. 복사한 이름을 그대로 사용해 브랜치를 만듭니다. Linear 이슈 ID(예: `TEAM-123`)를 제거하지 마세요.
4. GitHub에서 생성한 이슈가 Linear와 동기화되는 저장소라면 Linear에 같은 이슈를 다시 만들지 않습니다. 동기화된 Linear 이슈가 생성된 후 그 이슈의 브랜치 이름을 복사합니다.

브랜치 형식은 Linear의 GitHub integration 설정에서 팀 공통 형식으로 관리합니다.

## Pull Request

- 하나의 PR에는 하나의 논리적 변경만 담습니다.
- Linear 이슈를 완료하는 PR은 `Fixes TEAM-123`, 연결만 하는 PR은 `References TEAM-123` 형식으로 본문에 적습니다.
- 실행한 테스트와 확인 결과를 기록합니다.
- 호환성, 데이터 변경, 권한, 배포 순서 등 리뷰어가 확인해야 할 위험을 적습니다.
- 여러 저장소의 변경이 함께 필요하면 모든 관련 PR과 안전한 병합 순서를 연결합니다.

## 리뷰 기준

리뷰어는 다음을 중심으로 확인합니다.

- 변경이 이슈의 목적과 완료 조건을 충족하는가
- 기존 동작과 하위 호환성을 불필요하게 깨뜨리지 않는가
- 테스트 또는 검증 근거가 충분한가
- 운영, 보안, 개인정보, 데이터 마이그레이션 영향이 설명되어 있는가
- 관련 문서와 다른 저장소의 변경이 함께 반영되어 있는가
